## 3. 152-ФЗ — согласие и потоки 🔴 (главные дыры)

Инфраструктура согласия **построена, но не подключена** к регистрации, и нет UI приёма.

- 🔴 **Согласие при регистрации не сохраняется.** На `/register` есть обязательный непредзаполненный чекбокс согласия + необязательный чекбокс рекламной рассылки (`apps/web/src/app/register/page.tsx`), со ссылками на `/legal/consent` и `/legal/privacy`. НО backend-эндпоинт `POST /auth/register` (`apps/api/app/api/auth.py`) согласие **игнорирует** — запись `UserConsent` при регистрации не создаётся, выбор по рассылке не сохраняется. Итог: **нет доказательства согласия** (метка времени/IP/версия) на момент регистрации.
  → **Нужно:** в `/auth/register` принимать `consent_accepted` + `marketing_accepted` и создавать запись `UserConsent(consent_type="personal_data_processing", version, ip_address, accepted)` в той же транзакции; выбор по рассылке сохранять отдельным типом согласия/полем.
- 🔴 **Нет фронт-страницы `/consent`.** Механизм enforce (`check_consent_accepted`, `apps/api/app/api/consent.py`) при отсутствии согласия отдаёт `403 {redirect:"/consent"}` — но страницы `/consent` **не существует** (`apps/web/src/app/consent/` отсутствует). Реальный новый пользователь: регистрируется → идёт в тренировку → 403 → редирект в никуда. Демо не ловят это, т.к. их согласие засеяно.
  → **Нужно:** либо (а) писать согласие прямо при регистрации (тогда 403-петля не срабатывает) — предпочтительно; либо (б) построить страницу `/consent` с чекбоксом + `POST /consent/`.
- 🟠 **IP согласия за прокси, вероятно, неверный.** `accept_consent` пишет `ip_address = request.client.host`. За host-nginx это IP nginx-контейнера, если uvicorn не запущен с `--forwarded-allow-ips` и nginx не шлёт `X-Real-IP`/`X-Forwarded-For` (сам код это признаёт: `config.py:992`). Записанный IP согласия может быть бесполезен как доказательство.
  → **Нужно:** проверить запуск uvicorn (`--proxy-headers --forwarded-allow-ips=*`) и заголовки в nginx-vhost; тогда `request.client.host` = реальный IP.
- 🟢 **Механизм записи существует и корректен:** модель `UserConsent` (`user_consents`: `consent_type`, `version`, `accepted`, `ip_address`, `created_at`), `POST /consent/`, `GET /consent/status`, dependency `check_consent_accepted`. Версионирование согласия есть (`REQUIRED_CONSENTS = personal_data_processing v1.0`).

## 4. 152-ФЗ — трансграничная передача / поручение обработки 🔴 (ядро риска для ИИ-платформы)

**Факт (проверено по коду + prod.env):** единственный внешний AI-хост в проде — **`api.navy`** (navy.api, OpenAI-совместимый агрегатор). Через него идёт **весь ИИ**:
- **LLM** (`NAVY_LLM_URL=https://api.navy/v1`) — туда уходит **контент диалога тренировки** (реплики, которые пишет пользователь, роль-плей «менеджер ↔ клиент»), плюс промпты с именами персонажей.
- **STT** (распознавание речи) — `stt_provider=whisper`, ключи прямого Deepgram/OpenAI в prod.env **не заданы** → распознавание идёт через navy-прокси (`/v1/audio/transcriptions`). То есть **голос пользователя** (звонок + микрофон в чате) уходит на api.navy.
- **TTS** — `ELEVENLABS_API_KEY` прямой в prod.env **не задан** → синтез идёт через navy (`/v1/audio/speech`). Уходит текст реплик ИИ-клиента (не ПДн пользователя, но внешний сервис).

**Проблема по 152-ФЗ:**
1. `api.navy` — внешний обработчик. Это **поручение обработки ПДн третьему лицу** (ст. 6 ч. 3 152-ФЗ): требуется договор поручения + раскрытие в Политике.
2. Если `api.navy` (или модели за ним: DeepSeek, whisper, TTS) размещены **вне РФ** — это **трансграничная передача** (ст. 12): нужно отдельное информированное согласие субъекта + учитывать статус страны; данные о гражданах РФ по ст.18 ч.5 должны сначала записываться в БД в РФ (это соблюдается — Postgres/Redis локальные), но передача контента ИИ — отдельный вопрос.
3. **Тексты политики этому противоречат:** «не обрабатывает биометрию» (а голос шлётся), «не передаёт третьим лицам» (а контент уходит на api.navy).

**Что нужно (выбрать путь, это юр-решение владельца):**
- 🔴 Раскрыть в Политике ПДн: перечень привлекаемых обработчиков (навигация данных к ИИ-сервису), цель, объём; при иностранной юрисдикции — раздел о трансграничной передаче + **отдельное согласие** на неё при регистрации (ещё один `consent_type`).
- 🔴 Получить от провайдера `api.navy` **письменное подтверждение юрисдикции и размещения** обработки (РФ или нет) + договор поручения. Если РФ — риск снимается почти полностью; если нет — нужна трансграничная схема.
- 🟠 Технически минимизировать ПДн в промптах (не слать реальные ПДн пользователя в LLM; в роль-плее пользователь и так не вводит свои ПДн — это профессиональный диалог, что снижает риск, но голос остаётся).
- 🟠 Уточнить формулировки политики про биометрию/голос и про обработчиков.

**Оценка:** сам роль-плей обычно не содержит ПДн самого пользователя (он отыгрывает юриста, а не вводит свои паспортные данные), что снижает тяжесть. Но **голос** + **факт нераскрытой передачи** — это то, что нужно закрыть текстом + согласием + договором поручения.

## 5. 152-ФЗ — права субъекта, хранение, удаление 🟠

- 🟠 **Нет self-service удаления аккаунта.** В `apps/api/app/api/users.py` есть удаление аватара и друзей, но **нет удаления аккаунта/данных**. Политика (разд. 8) предлагает отзыв/удаление по письму на hello@ — формально по 152-ФЗ допустимо (по обращению), но это ручной процесс. Рекомендуется добавить кнопку «Удалить аккаунт» в настройках (danger zone) с реальным удалением/обезличиванием.
- 🟠 **Нет retention/cleanup-джобы.** Нет автоматической очистки старых транскриптов/сессий/ПДн (grep не нашёл retention/purge/TTL по ПДн). Политика обещает «после достижения целей данные удаляются или обезличиваются» — но механизма нет → **расхождение обещания и реальности** (хранение бессрочно). Нужна политика хранения + джоба (напр. удалять/обезличивать транскрипты старше N мес.).
- 🟢 **Целостность при удалении заложена:** `training_sessions → users ondelete=CASCADE`, `messages → training_sessions CASCADE` (`apps/api/app/models/training.py`). При удалении пользователя его сессии/сообщения каскадно удалятся — но триггера удаления нет.
- 🟢 **Локализация данных в РФ** — Postgres/Redis — локальные контейнеры на московской VM. Заявлено в политике (разд. 5) и соответствует факту.
