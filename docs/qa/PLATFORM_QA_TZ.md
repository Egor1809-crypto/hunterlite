# ТЗ: ручная проверка каждой функции HunterLite

> Документ для ручного тестировщика. Цель — пройти **все** пользовательские функции платформы и зафиксировать, что каждая работает. Платформа боевая (prod), любые действия пишут реальные данные. Действуй аккуратно, не нагружай прод спамом.

## Как пользоваться

- **Прод-URL:** https://legalhunter.pro
- **Демо-аккаунты** (пароли регистрозависимы, `N` — это цифра 1..5):

  | Логин | Пароль | Роль | План |
  |---|---|---|---|
  | `demo1@hunterlite.ru` | `Demo1Pass!2026` | **admin** | master (по роли) |
  | `demo2@hunterlite.ru` | `Demo2Pass!2026` | **rop** (руководитель) | master (по роли) |
  | `demo3@hunterlite.ru` | `Demo3Pass!2026` | manager | scout/trial |
  | `demo4@hunterlite.ru` | `Demo4Pass!2026` | manager | scout/trial |
  | `demo5@hunterlite.ru` | `Demo5Pass!2026` | manager | scout/trial |

- **ВАЖНО — состояние демо-аккаунтов (проверено по `seed_db.py` / `entitlement.py` / `seed_demo_progress.py`):**
  - Роли РАЗНЫЕ (специально, чтобы проверять разграничение доступов). НЕ «все админы».
  - **План `master` получают только demo1 (admin) и demo2 (rop)** — через автоповышение `ELEVATED_ROLES = {admin, rop}`. **demo3/4/5 (managers) — план `scout` с 14-дневным триалом**, у них работают лимиты плана (дневные лимиты тренировок/PvP/RAG). Это значит: проверку лимитов плана делать на demo3/4/5, проверку «безлимита» — на demo1/demo2.
  - **152-ФЗ согласие уже принято** для всех демо (засеяно), поэтому экран `ConsentGate` им показываться не должен. У реального нового пользователя UI приёма согласия пока НЕ построен (продуктовый гэп) — это отдельная заметка, не баг демо.
  - **Конструктор персонажей разблокирован** (регион-1 карты тестов засеян как пройденный 100%) — у всех 5 демо.
  - `email_verified` в БД = `false`, но **на логине не проверяется** → вход не блокируется. Не считать «не верифицирован» багом демо.
  - `must_change_password = false` → демо НЕ должны попадать на экран принудительной смены пароля.

- **Формат чек-листа:** `- [ ]` шаг → **Ожидаемо:** наблюдаемый результат.
- **Двухаккаунтные проверки** помечены **[2 аккаунта]** — нужны два параллельных входа (напр. два браузера/инкогнито).
- Если шаг помечен **[admin]** / **[rop]** — выполнять под demo1 / demo2 соответственно.
- Прогоняй на десктопе И на мобильном (ширина < 1024px) там, где это указано.

## 1. Аутентификация и онбординг

Маршруты: `/login`, `/register`, `/reset-password`, `/change-password`, `/auth/callback`. Логика входа/обновления токена — `lib/api.ts`, `AuthLayout`, `PageAuthGate`.

### 1.1 Вход (`/login`)
- [ ] Открыть `/login`, ввести `demo1@hunterlite.ru` / `Demo1Pass!2026`, нажать **Войти**. → **Ожидаемо:** переход на `/home`, в шапке имя «Алексей Иванов».
- [ ] Ввести email с верхним регистром (`DEMO1@hunterlite.ru`). → **Ожидаемо:** вход проходит (email нормализуется/триммится).
- [ ] Оставить email пустым / ввести `abc` (без `@`). → **Ожидаемо:** красный блок «Введите корректный email», запрос не уходит.
- [ ] Ввести пароль из 1–3 символов. → **Ожидаемо:** «Пароль должен содержать минимум 4 символа» (на логине минимум 4, НЕ 8).
- [ ] Ввести верный email и НЕВЕРНЫЙ пароль. → **Ожидаемо:** красный блок с ошибкой сервера (неверные данные), без перехода.
- [ ] **Лимит попыток:** быстро отправить форму входа 16+ раз подряд (лимит 15/мин). → **Ожидаемо:** появляется текст «Слишком много запросов. Подождите N секунд.» (отдельного экрана блокировки нет, это сообщение из `lib/api.ts`).
- [ ] Начать вводить email, перезагрузить страницу. → **Ожидаемо:** email подставлен обратно (хранится в `sessionStorage["vh-login-email"]`); после успешного входа ключ очищается.
- [ ] Нажать «глаз» в поле пароля. → **Ожидаемо:** пароль показывается/скрывается, кнопка не попадает в таб-навигацию.
- [ ] Кнопка **Зарегистрироваться** внизу. → **Ожидаемо:** переход на `/register`.
- [ ] **OAuth-кнопки:** проверить, что кнопка **Yandex** и разделитель «или» показываются (provider сконфигурирован). → **Ожидаемо:** видна кнопка «Yandex»; кнопки Google быть НЕ должно (149-ФЗ).
- [ ] **Yandex-вход:** нажать «Yandex». → **Ожидаемо:** редирект на `oauth.yandex.ru`, после согласия — возврат на `/auth/callback` → «Вход выполнен!» → `/home`. (Нужен реальный Yandex-аккаунт.)
- [ ] **Open-redirect:** открыть `/login?redirect=//evil.com`, войти. → **Ожидаемо:** редирект игнорируется, попадаем на `/home` (не на внешний домен).
- [ ] **Сетевой сбой:** отключить сеть, нажать Войти. → **Ожидаемо:** полноэкранный «рыбак» `FishermanError` («Похоже, сервер временно недоступен…») с кнопкой «Переподключиться» (НЕ обычный inline-текст).

### 1.2 Восстановление пароля (внутри `/login`)
- [ ] Нажать **Забыли пароль?**. → **Ожидаемо:** форма флипается в под-вид (без перехода), email подставлен из логина.
- [ ] Ввести email, нажать **Отправить ссылку**. → **Ожидаемо:** экран «Письмо отправлено» с зелёной галочкой и указанным email.
- [ ] Ввести заведомо НЕсуществующий email и отправить. → **Ожидаемо:** всё равно показывается «Письмо отправлено» (защита от перебора — успех всегда).
- [ ] Кнопка **Назад**. → **Ожидаемо:** возврат к форме входа.

### 1.3 Регистрация (`/register`)
- [ ] Открыть `/register`, заполнить Полное имя, Email, Пароль, Повтор. → **Ожидаемо:** при валидных данных — переход на `/home` (создаётся новый аккаунт). *Использовать одноразовый email, не демо.*
- [ ] Оставить имя пустым. → **Ожидаемо:** «Укажите имя».
- [ ] Ввести разные пароль и повтор. → **Ожидаемо:** живая красная подсказка «Пароли не совпадают.», кнопка регистрации заблокирована.
- [ ] Проверить **чек-лист пароля** (5 правил): по мере ввода зелёными становятся — Минимум 8 символов / Заглавная / Строчная / Цифра / Спецсимвол. → **Ожидаемо:** все 5 загораются; пока не все — submit даёт «Пароль не соответствует требованиям безопасности».
- [ ] Кнопка **Войти** внизу. → **Ожидаемо:** переход на `/login`.

### 1.4 Сброс пароля по ссылке (`/reset-password`)
- [ ] Открыть `/reset-password` БЕЗ `?token=`. → **Ожидаемо:** мгновенный редирект на `/` (страница не рендерится).
- [ ] Открыть с валидным токеном (из письма), задать новый пароль (чек-лист 5 правил) + подтверждение, нажать **Сохранить пароль**. → **Ожидаемо:** зелёный экран «Пароль изменён!», авто-редирект на `/` через ~3 сек.
- [ ] Открыть с устаревшим/использованным токеном. → **Ожидаемо:** «Ссылка устарела или уже использована. Запросите новую.»

### 1.5 Принудительная смена пароля (`/change-password`)
- [ ] (Под демо НЕ воспроизводится, т.к. `must_change_password=false`.) Если попали сюда — задать новый пароль, **Сменить пароль**. → **Ожидаемо:** переход на `/home`.

### 1.6 OAuth callback (`/auth/callback`)
- [ ] Пройти вход через Yandex. → **Ожидаемо:** экран «АВТОРИЗАЦИЯ…» → «Вход выполнен!» → редирект на `/home` (~0.8 сек).
- [ ] Открыть `/auth/callback` без `?code=`. → **Ожидаемо:** «Ошибка входа / Код авторизации не получен» + кнопка «Вернуться к входу».

### 1.7 Сессия и авто-обновление токена (cross-cutting)
- [ ] Войти, затем **жёстко перезагрузить** `/home` (Ctrl+R). → **Ожидаемо:** остаёмся залогинены (тихий `POST /auth/refresh` по куке `vh_authenticated`), НЕ выкидывает на `/login`.
- [ ] Очистить in-memory токен (просто долго не трогать вкладку, затем вернуться) на странице тренировки/квиза. → **Ожидаемо:** `PageAuthGate` либо восстанавливает сессию, либо ведёт на `/login` (без белого экрана).
- [ ] **Выход:** меню пользователя (сайдбар) → **Выйти**. → **Ожидаемо:** редирект на `/`, повторный заход на `/home` требует логина.
- [ ] Ошибка подключения при бутстрапе (отключить API) → **Ожидаемо:** карточка «Ошибка подключения» (`BootErrorCard`) с «Повторить» (до 5 ретраев с backoff, потом «Слишком много попыток. Перезагрузите страницу.»).

## 3. Тренировка — чат (WebSocket)

Маршруты: `/training` (лаунчер), `/training/[id]` (живая сессия). Транспорт: `/ws/training` (аутентификация первым WS-сообщением, лимит 60 сообщ./10 сек, повторная проверка плана каждые 5 мин).

### 3.1 Лаунчер (`/training`)
- [ ] Открыть `/training`. → **Ожидаемо:** заголовок «Обучение», виджет пути (5 этапов-чипов со %), сегмент-контрол «Тесты» / «Мои клиенты».
- [ ] Вкладка «Мои клиенты» (конструктор). → **Ожидаемо:** галерея reference-персонажей (≈25 карточек: архетип, профессия, сложность). У демо конструктор разблокирован — карточки видны (не «Практика пока закрыта»).
- [ ] Фильтры по сложности и архетипу. → **Ожидаемо:** список фильтруется; цвет — только сигнал сложности.
- [ ] Выбрать персонажа → превью → **Начать (чат)**. → **Ожидаемо:** `POST /training/sessions`, переход в `/training/{id}`.
- [ ] **Edge:** начать вторую тренировку, не закрыв первую. → **Ожидаемо:** тост «Активная тренировка» + редирект в уже идущую сессию (~0.6 сек) (409 `session_already_active`).

### 3.2 Живой чат (`/training/[id]`) — десктоп
- [ ] Войти в сессию. → **Ожидаемо:** экран «Подключение к сессии…», затем чат открывается сразу (без брифинга/мик-чека).
- [ ] Шапка: таймер по центру, «Завершить» справа. → **Ожидаемо:** таймер идёт; на 25:00 (1500с) цвет становится предупреждающим.
- [ ] Ввести сообщение, **Enter**. → **Ожидаемо:** оптимистичный пузырь «ВЫ», отправка `text.message`, приходит ответ клиента.
- [ ] **Shift+Enter** в поле ввода. → **Ожидаемо:** перенос строки, не отправка.
- [ ] Поле ввода — длинный текст. → **Ожидаемо:** авто-рост до 240px.
- [ ] **Индикатор печати:** после отправки. → **Ожидаемо:** «{имя} печатает...» с тремя точками; снимается по приходу ответа (или через 45с по watchdog).
- [ ] **Автоскролл:** при стриминге ответа. → **Ожидаемо:** чат держится у нижней кромки, докручивается на новых чанках.
- [ ] **Звук в чате:** → **Ожидаемо:** TTS в чате ВЫКЛЮЧЕН (ответы только текст) — отсутствие голоса в чате это норма, не баг.

### 3.3 Оценка и подсказки (правый сайдбар, десктоп)
- [ ] Переключатель «Оценка» / «Реакции». → **Ожидаемо:** «Оценка» — `VibeMeter`, доля речи (talk%), Общий балл /100 (анимация, «—» до первого сообщения), 5 осей (Полнота выяснения /18, Правовая точность ФЗ-127 — «после завершения», Корректность рекомендации /18, Отработка сомнений /12, Этические нарушения /15).
- [ ] «Реакции». → **Ожидаемо:** `LiveEmotionTimeline` (нужно ≥2 точек эмоций, иначе пусто).
- [ ] **Whisper-панель / подсказки:** тумблер «Подсказки (AI-коуч) Вкл/Выкл». → **Ожидаемо:** при Выкл — «Подсказки отключены»; при Вкл — подсказки/коучинг-замечания появляются сверху списком (новые выше, старые блёкнут).
- [ ] Тумблер «Варианты ответа». → **Ожидаемо:** до 3 карточек-вариантов (`/training/sessions/{id}/script-hints`); **тап вставляет текст в поле ввода** и фокусирует его.
- [ ] Кликнуть whisper типа «Этап N». → **Ожидаемо:** скроллит и раскрывает ScriptPanel.

### 3.4 Голосовой ввод в чате (микрофон)
- [ ] Тап по кнопке микрофона. → **Ожидаемо:** старт записи (кнопка красная, иконка «квадрат», полоса «Идёт запись…» с волной уровня). Повторный тап — стоп + распознавание (это тап-тоггл, НЕ зажатие).
- [ ] После записи. → **Ожидаемо:** распознанный текст попадает в поле как редактируемый превью («Распознано — проверьте и отправьте»), подтверждение — Enter/отправка.
- [ ] **Edge:** очень короткий тап (<2КБ). → **Ожидаемо:** тост «Слишком короткое нажатие», запрос не уходит.
- [ ] **Edge:** запретить доступ к микрофону в браузере. → **Ожидаемо:** тост «Микрофон недоступен» с инструкцией, режим переключается на текст (НЕ зависает).
- [ ] **Edge (watchdog):** если распознавание не пришло 45с. → **Ожидаемо:** индикатор «Распознаю…» сбрасывается + предупреждающий тост (не виснет навсегда).

### 3.5 Завершение и спец-модалки
- [ ] Нажать **Завершить** → модалка «Завершить консультацию?» → подтвердить. → **Ожидаемо:** `session.end`, переход на `/results/{id}` (фолбэк-редирект через 5 сек, если событие не пришло).
- [ ] Написать прощание («до свидания», «кладу трубку»). → **Ожидаемо:** открывается HangupModal.
- [ ] **Молчание:** ничего не писать долго. → **Ожидаемо:** баннер `silence.warning`, затем модалка «Вы давно молчите» (Продолжить / Завершить).
- [ ] **Клиент бросает трубку:** довести клиента до срыва. → **Ожидаемо:** тост «Клиент теряет терпение…», затем HangupModal с причиной; если продолжить нельзя — авто-`session.end` через 3 сек, кнопки «Перезвонить» / «К результатам».
- [ ] **Escape** на экране чата. → **Ожидаемо:** тоггл модалки прерывания.

### 3.6 Мобильный режим чата (ширина < 1024px) — «остров»
- [ ] Открыть `/training/[id]` на мобиле. → **Ожидаемо:** весь экран — лента чата, снизу плавающий полупрозрачный «остров»; свёрнутый = бар ввода + чип персоны (имя, эмоция, балл, индикатор речи).
- [ ] Написать сообщение в свёрнутом острове, отправить. → **Ожидаемо:** ответ приходит, лента автоскроллится вниз, последнее сообщение не под островом.

## Приложение: известные особенности (НЕ баги)

- Минимальная длина пароля на **логине = 4**, при регистрации/сбросе/смене/в настройках — полная политика (≥8 + сложность). Пароль 4–7 символов может войти, но не может быть установлен.
- В чате TTS выключен (текст), в звонке и у Маняши-квиза — тоже (только звонок-режим `/call` озвучивает клиента).
- Карта тестов: порог уровня **88%**; попытки/энергия списываются на ЗАВЕРШЕНИИ, не на старте; пересдача пройденного уровня бесплатна.
- Коды закрытия WS 4001/4002/4003 намеренно НЕ переподключаются.
- Нет cookie-баннера и нет глобального offline-баннера — только 152-ФЗ `ConsentGate`, `FishermanError` и `LLMDegradationBanner`.
- `/exam/certificate` (без examId) — 404 (страницы нет).
- Вход: собственный email/пароль + **Yandex ID** (149-ФЗ). Google убран. **SMS-вход ещё не реализован** (по ТЗ `docs/auth/AUTH_REBUILD_TZ.md`).
- Роутеры `arena_lifelines`/`arena_powerups` не смонтированы (мёртвый PvP-код) — соответствующие REST-вызовы дадут 404; в проверку не входят.
