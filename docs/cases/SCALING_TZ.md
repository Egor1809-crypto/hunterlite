# ТЗ: масштабирование кейсов БФЛ до 50 (для агента-исполнителя)

> **Статус эталона:** кейс `bfl-01` (Сергей: кредиты после потери работы) ПОЛНОСТЬЮ
> одобрен заказчиком. Это референс. Все 49 новых кейсов делаются строго по той же
> механике, контракту данных и стилю. Бэкенд и фронтенд менять НЕ нужно — вы
> добавляете только ДАННЫЕ (контент кейсов) в seed-слой.

> **Материал — ТОЛЬКО реальная судебная практика банкротства физических лиц (БФЛ)
> по ФЗ-127 «О несостоятельности (банкротстве)», гл. X §1.1 и §5, ст. 213.x, и
> смежным нормам (ГПК ст. 446, ГК).** Никаких вымышленных «процедур». Никакой
> путаницы с банкротством юридических лиц (наблюдение, внешнее управление,
> конкурсное производство, КДЛ, субсидиарка ЮЛ — НЕ ИСПОЛЬЗОВАТЬ как «правильные»
> шаги; они допустимы только как дистракторы с пометкой «это про юрлиц»).

## 0. TL;DR для исполнителя

1. Кейсы живут одной строкой на кейс в таблице `case_scenarios` (PostgreSQL, JSONB).
2. Их сеет один скрипт: `apps/api/scripts/seed_cases.py` → список `CASES`.
3. Ваша работа: довести `CASES` с 1 до 50 элементов, каждый — по структуре `CASE_BFL_01`.
4. Каждый кейс = 100 баллов: **Этап 1** (дерево из 5 вопросов × 5 вариантов, 50 б.) +
   **Этап 2** (хронология: 7 верных шагов + 5 дистракторов, 50 б.).
5. После каждого батча: прогнать валидатор (§9), посеять, проверить в браузере (§10).
6. Готово = валидатор зелёный + кейс проходится на 100% при верных ответах + дерево
   реально ветвится + юридически достоверно.

## 2. Контракт данных (JSONB) — копировать дословно

Бэкенд `apps/api/app/api/cases.py` читает строго эти поля. Лишние поля игнорируются,
отсутствие обязательных = сломанный кейс. Ниже — единственно допустимая форма.

### 2.1 `stage1` — дерево решений

```python
"stage1": {
    "intro": "Короткий вводный текст (1–2 предложения).",
    "start": "q1",                       # id стартового узла (всегда вопрос)
    "correct_path": ["q1-a","q2-a","q3-a","q4-a","q5-a"],   # id верных option (документирующее поле)
    "correct_outcome": "out-...",        # id исхода (документирующее поле)
    "nodes": {                           # СЛОВАРЬ всех узлов по их id
        # ── узлы типа question (их РОВНО 5 — это «спина») ──
        "q1": {
            "type": "question",
            "step": 1,                   # 1..5, номер шага спины (для прогресс-бара)
            "title": "Шаг 1. Короткий заголовок",
            "question": "Текст вопроса?",
            "facts": [                   # 2–4 факт-карточки (визуализация)
                {"label": "Долг", "value": "1 300 000 ₽"},
                {"label": "Лимит МФЦ", "value": "25 000 – 1 000 000 ₽"}
            ],
            "options": [                 # РОВНО 5 вариантов
                {"id":"q1-a","text":"...","correct": True,  "next":"q2",            "explain":"Короткий вердикт (1 предл.)"},
                {"id":"q1-b","text":"...","correct": False, "next":"info_x",        "explain":"Короткий вердикт почему неверно"},
                {"id":"q1-c","text":"...","correct": False, "next":"info_y",        "explain":"..."},
                {"id":"q1-d","text":"...","correct": False, "next":"info_z",        "explain":"..."},
                {"id":"q1-e","text":"...","correct": False, "next":"info_w",        "explain":"..."}
            ]
        },
        # ... q2..q5 аналогично, q5 верный option ведёт next → исход "out-..."

        # ── узлы типа info (разбор ошибки) ──
        "info_x": {
            "type": "info",
            "step": 1,                   # к какому шагу относится разбор
            "title": "Заголовок разбора",
            "body": "Развёрнутое объяснение (2–4 предложения), реальная практика.",
            "facts": [ {"label":"...","value":"..."} ],   # 0–3 факта (можно [])
            "next": "q2"                 # КУДА сходится после разбора (следующий вопрос спины или исход)
        },

        # ── узел типа outcome (исход этапа 1) ──
        "out-...": {
            "type": "outcome",
            "outcome": "realization",    # машинный код исхода (свободный slug)
            "title": "Заголовок исхода",
            "summary": "Что решили и что дальше (3–5 предложений). Мост ко 2 этапу."
        }
    }
}
```

**Правила ветвления (ОБЯЗАТЕЛЬНЫ):**

1. Ровно **5** узлов `type:"question"` (`q1…q5`), у каждого `step` = 1…5 соответственно.
2. У каждого вопроса **ровно 5** `options`, среди них **ровно 1** `correct: True`.
3. **Верный** option ведёт `next` → следующий вопрос спины (`q5` верный → `out-...` исход).
4. **Каждый неверный** option ведёт `next` → узел `type:"info"` (разбор), а info-узел `next`
   сходится вперёд (к следующему вопросу спины или к исходу). Линейного «неверно → сразу
   тот же вопрос» быть не должно — это полноценное дерево.
5. **Пересечения веток (по ТЗ):** соседние/родственные ошибки ДОЛЖНЫ делить общие info-узлы.
   Минимум **3 общих info-узла на кейс** (один info — `next` от ≥2 разных неверных option,
   в т.ч. желательно из разных вопросов). Пример из `bfl-01`: `info_wait_bank` ← `q1-e` и `q3-b`.
6. Все `id` в `next` обязаны существовать в `nodes`. Нет висячих ссылок. Нет циклов назад
   (граф — DAG, всегда вперёд к исходу).
7. `option.id` уникальны в пределах кейса, формат `q<N>-<a..e>`. `info`-узлы: `info_<slug>`.
8. `explain` у option — короткий (1 предложение) вердикт. Глубокий разбор — в `info.body`.

**Что отдаётся клиенту (для понимания, менять бэкенд НЕ надо):**
- `question` отдаётся без `correct`/`next`/`explain` (стрипается). `step`,`title`,`question`,`facts`,`options{id,text}` — да.
- `info` отдаётся с вшитым уже разрешённым `next` (клиент идёт дальше без доп. запроса).
- Скоринг: `stage1_score = (число верно отвеченных вопросов) × 10`. Только вопросы считаются
  (info не «отвечаются»). 5 верных = 50 баллов.

### 2.2 `stage2` — хронология процедуры

```python
"stage2": {
    "prompt": "Что нужно сделать (1–2 предложения).",
    "correct_sequence": ["s1","s2","s3","s4","s5","s6","s7"],   # РОВНО 7 верных шагов в правильном порядке
    "pool": [
        # 7 ВЕРНЫХ (is_correct=True, order=1..7, порядок order совпадает с местом в correct_sequence)
        {"id":"s1","text":"...","is_correct": True,  "order": 1, "explain":"Почему этот шаг и почему здесь."},
        {"id":"s2","text":"...","is_correct": True,  "order": 2, "explain":"..."},
        {"id":"s3","text":"...","is_correct": True,  "order": 3, "explain":"..."},
        {"id":"s4","text":"...","is_correct": True,  "order": 4, "explain":"..."},
        {"id":"s5","text":"...","is_correct": True,  "order": 5, "explain":"..."},
        {"id":"s6","text":"...","is_correct": True,  "order": 6, "explain":"..."},
        {"id":"s7","text":"...","is_correct": True,  "order": 7, "explain":"..."},
        # 5 ДИСТРАКТОРОВ (is_correct=False, order=None) — правдоподобные, но неверные шаги
        {"id":"d1","text":"...","is_correct": False, "order": None, "explain":"Почему это НЕ этап / ошибка."},
        {"id":"d2","text":"...","is_correct": False, "order": None, "explain":"..."},
        {"id":"d3","text":"...","is_correct": False, "order": None, "explain":"..."},
        {"id":"d4","text":"...","is_correct": False, "order": None, "explain":"..."},
        {"id":"d5","text":"...","is_correct": False, "order": None, "explain":"..."}
    ]
}
```

**Правила хронологии (ОБЯЗАТЕЛЬНЫ):**

1. Ровно **7** верных (`is_correct:True`, `order` 1…7 без пропусков и дублей) + ровно **5**
   дистракторов (`is_correct:False`, `order:None`). Итого пул = **12** элементов.
2. `correct_sequence` = `["s1"…"s7"]` в порядке, СОВПАДАЮЩЕМ с `order`. (Сортировка верных по
   `order` обязана давать тот же список.)
3. `id` уникальны (`s1..s7`, `d1..d5`). У каждого элемента — содержательный `explain`.
4. Дистракторы — из реальных заблуждений: путь через МФЦ когда нельзя, «договориться с банком о
   списании», сокрытие/переоформление имущества, новый кредит, ожидание исковой давности, шаги
   из банкротства ЮЛ (наблюдение/внешнее управление — пометить, что это про юрлиц) и т.п.
5. Скоринг: `stage2_score = round(50 × matched / 7)`, где `matched` — число верных, стоящих на
   своей позиции. 7/7 = 50 баллов. Поэтому `len(correct_sequence)` обязан быть 7.

### 2.3 Прочие поля кейса

- `expert_analysis` — итоговый разбор (показывается на экране результата). 5–9 предложений:
  какой путь и почему, судьба имущества, ключевые нормы, предупреждение о добросовестности.
- `max_score` — всегда `100` (50+50). Не менять.

## 3. Инварианты методики (чек-лист на каждый кейс)

- [ ] 1 герой, 1 правдоподобная жизненная фабула (реальная практика).
- [ ] Этап 1: 5 вопросов × 5 вариантов; в каждом 1 верный.
- [ ] Каждый неверный вариант → info-разбор → сходится вперёд.
- [ ] ≥3 общих info-узла (пересечения веток), желательно между разными вопросами.
- [ ] 2–4 `facts` на каждом вопросе; у части info — свои `facts`.
- [ ] Исход (`outcome`) логично вытекает из верного пути и ведёт ко 2 этапу.
- [ ] Этап 2: 7 верных шагов (правильный хронологический порядок) + 5 дистракторов.
- [ ] `expert_analysis` ёмкий и юридически точный.
- [ ] Текст — в фирменном стиле (см. §8): сухо, по-деловому, без воды и канцелярита-каши.
- [ ] Нигде нет ассистента «Маняши», нет про-юрлицо терминов как «правильных».

## 4. Юридические требования (реальная практика БФЛ)

**Опорные нормы и факты (актуализируй на дату исполнения — законы меняются):**

- **Внесудебное банкротство (МФЦ), ст. 223.2 ФЗ-127:** сумма обязательств
  **25 000 – 1 000 000 ₽**; базовое основание — оконченное ИП по «отсутствию имущества»
  (п. 4 ч. 1 ст. 46 ФЗ-229). Расширенные основания (2023): пенсионер/получатель пособий с
  единственным доходом и взысканием > 1 года; исполнение > 7 лет. Срок процедуры — 6 мес.
- **Судебное банкротство:** заявление в **арбитражный суд** по месту жительства.
  **Обязанность** должника подать — при долге **≥ 500 000 ₽** и просрочке **> 3 мес** (30 раб. дней);
  **право** — при любой сумме, если очевидна неплатёжеспособность.
- **Госпошлина** по делу о банкротстве гражданина и **депозит** на вознаграждение финансового
  управляющего (фикс. часть 25 000 ₽ за процедуру + 7% от удовлетворённых требований/выручки).
  Конкретную сумму госпошлины — СВЕРИТЬ на дату (менялась); в `bfl-01` указано 10 000 ₽.
- **Реструктуризация долгов:** план до **5 лет**, требует подтверждённого регулярного дохода,
  достаточного для графика и содержания должника/иждивенцев.
- **Реализация имущества:** конкурсная масса, опись/оценка, **электронные торги**; вводится при
  отсутствии дохода/невозможности плана. По итогу — освобождение от долгов.
- **Мировое соглашение:** на любой стадии, с согласия кредиторов, прекращает процедуру.
- **Исполнительский иммунитет (ст. 446 ГПК):** единственное пригодное жильё (НЕ в ипотеке/залоге),
  личные вещи, проф. инструмент в пределах лимита, прожиточный минимум должнику и иждивенцам.
  Практика КС РФ (Пост. № 11-П/2021) — «роскошное» единственное жильё может замещаться.
- **Ипотечное единственное жильё:** в залоге — иммунитета НЕТ, идёт в массу залоговому кредитору
  (80% выручки). Учесть закон 2024 о возможности сохранить ипотечное единственное жильё через
  мировое/локальный план с залоговым кредитором — отдельный архетип.
- **Оспаривание сделок:** ст. **61.2** (подозрительные — дарение/занижение цены родне за 1–3 года),
  ст. **61.3** (предпочтение одному кредитору за 1–6 мес). Последствие — возврат в массу.
- **Недобросовестность → НЕ освобождают от долгов** (п. 4 ст. 213.28): сокрытие имущества,
  предоставление заведомо ложных сведений банку при получении кредита, злостное уклонение.
- **Долги, НЕ списываемые** (п. 5–6 ст. 213.28): алименты; возмещение вреда жизни/здоровью и
  морального вреда; зарплата/выходные пособия (если должник — работодатель); субсидиарная
  ответственность; возмещение по умышленному/неосторожному причинению; текущие платежи.
- **Совместное имущество супругов:** реализуется общее, доля супруга возвращается деньгами;
  возможно совместное банкротство (практика ВС).
- **Последствия:** 5 лет указывать на банкротство при кредитах; 3 года нельзя руководить ЮЛ
  (10 лет — банк, 5 — страховая/МФО/НПФ); повторное банкротство — через 5 лет.

**Запрещено (частые ошибки авторов):**
- Тащить термины ЮЛ-банкротства как верные шаги: наблюдение, финансовое оздоровление, внешнее
  управление, конкурсное производство, собрание кредиторов «как у ЮЛ», КДЛ. (Можно — как
  дистрактор с пометкой «это процедуры юрлиц».)
- Путать районный суд и арбитражный (банкротство граждан — арбитражный).
- Обещать «суд сразу спишет без процедуры», «банк сам всё спишет по просьбе».
- Устаревшие/выдуманные цифры лимитов и пошлин — сверять.

## 5. Таксономия: категории, сложность, нумерация

**ID / порядок / код**
- `id`: `bfl-01` … `bfl-50` (стабильно, ≤32 симв.).
- `order_index`: `0` … `49`, плотно и уникально. Код в UI = `БФЛ_{order_index+1:02d}`.
- `bfl-01` уже занят (эталон) — новые с `bfl-02` / `order_index=1`.

**Категории (`category`, ≤128 симв.) — пул значений:**
`Внесудебное (МФЦ)` · `Реализация имущества` · `Реструктуризация` · `Мировое соглашение` ·
`Ипотека и жильё` · `Имущество супругов` · `Оспаривание сделок` · `Недобросовестность` ·
`Несписываемые долги` · `Залоговый кредитор` · `Поручительство` · `ИП-гражданин` ·
`Микрозаймы и МФО` · `Последствия банкротства`.

**Difficulty 1…5** (растёт с числом ловушек/нюансов):
1 — базовый выбор пути; 2 — один нюанс (имущество/доход); 3 — два-три нюанса; 4 — спорные
сделки/иммунитет/супруги; 5 — пограничная практика (КС, несписываемые, совместное банкротство).

Распределение по 50 кейсам (ориентир): D1 ×8, D2 ×12, D3 ×14, D4 ×10, D5 ×6.

## 6. Каталог 50 архетипов (материал — реальная практика)

`bfl-01` готов. Сделать `bfl-02`…`bfl-50` по списку ниже. Каждый — самостоятельная фабула;
«изюминка» = что именно проверяет дерево. Можно адаптировать имена/суммы, но юридический
смысл и исход — строго по практике.

## 9. Валидатор (запускать перед каждым посевом)

Создай `apps/api/scripts/validate_cases.py` со следующими проверками (offline, без БД).
Скрипт обязан падать с понятным сообщением на первой же ошибке.

```python
"""Структурная валидация кейсов перед посевом. Запуск: python -m scripts.validate_cases"""
from scripts.cases_data import CASES  # или: from scripts.seed_cases import CASES

def err(cid, msg): raise SystemExit(f"[{cid}] {msg}")

def validate(case):
    cid = case["id"]
    # — обязательные поля верхнего уровня —
    for f in ("id","title","description","difficulty","category","estimated_minutes",
              "max_score","expert_analysis","stage1","stage2","order_index"):
        if f not in case: err(cid, f"нет поля {f}")
    if len(case["id"]) > 32: err(cid, "id > 32 символов")
    if len(case["title"]) > 256: err(cid, "title > 256")
    if len(case["category"]) > 128: err(cid, "category > 128")
    if case["max_score"] != 100: err(cid, "max_score должен быть 100")
    if case["difficulty"] not in (1,2,3,4,5): err(cid, "difficulty 1..5")

    s1 = case["stage1"]; nodes = s1["nodes"]
    if s1["start"] not in nodes: err(cid, "start отсутствует в nodes")
    questions = {k:v for k,v in nodes.items() if v.get("type")=="question"}
    infos     = {k:v for k,v in nodes.items() if v.get("type")=="info"}
    outcomes  = {k:v for k,v in nodes.items() if v.get("type")=="outcome"}
    if len(questions) != 5: err(cid, f"вопросов {len(questions)}, нужно 5")
    if len(outcomes) < 1: err(cid, "нет исхода (outcome)")
    if sorted(v["step"] for v in questions.values()) != [1,2,3,4,5]:
        err(cid, "step вопросов должны быть 1..5 без дублей")

    info_refs = {}  # сколько ссылок ведёт в каждый info (для проверки пересечений)
    for qid,q in questions.items():
        opts = q["options"]
        if len(opts) != 5: err(cid, f"{qid}: вариантов {len(opts)}, нужно 5")
        if sum(1 for o in opts if o.get("correct")) != 1:
            err(cid, f"{qid}: верный вариант должен быть ровно 1")
        for o in opts:
            if not o.get("text") or not o.get("explain"): err(cid, f"{qid}/{o.get('id')}: пустой text/explain")
            nx = o.get("next")
            if nx not in nodes: err(cid, f"{qid}/{o['id']}: next '{nx}' не найден")
            tgt = nodes[nx]
            if o.get("correct"):
                if tgt["type"] not in ("question","outcome"):
                    err(cid, f"{qid}/{o['id']}: верный должен вести в вопрос/исход")
            else:
                if tgt["type"] != "info":
                    err(cid, f"{qid}/{o['id']}: неверный должен вести в info")
                info_refs[nx] = info_refs.get(nx,0)+1
        if not (2 <= len(q.get("facts",[])) <= 4): err(cid, f"{qid}: нужно 2..4 facts")

    for iid,info in infos.items():
        if not info.get("body"): err(cid, f"{iid}: пустой body")
        if info.get("next") not in nodes: err(cid, f"{iid}: next не найден")
        if nodes[info["next"]]["type"] not in ("question","outcome"):
            err(cid, f"{iid}: info должен сходиться в вопрос/исход")
    # пересечения: минимум 3 info, в которые ведут ≥2 разных неверных option
    shared = [i for i,c in info_refs.items() if c >= 2]
    if len(shared) < 3:
        err(cid, f"пересечений веток мало: общих info {len(shared)}, нужно ≥3")

    # — stage2 —
    s2 = case["stage2"]; pool = s2["pool"]; seq = s2["correct_sequence"]
    correct = [p for p in pool if p.get("is_correct")]
    distr   = [p for p in pool if not p.get("is_correct")]
    if len(correct) != 7: err(cid, f"верных шагов {len(correct)}, нужно 7")
    if len(distr) != 5: err(cid, f"дистракторов {len(distr)}, нужно 5")
    if len(seq) != 7: err(cid, "correct_sequence должен быть длиной 7")
    if sorted(p["order"] for p in correct) != [1,2,3,4,5,6,7]:
        err(cid, "order верных шагов должен быть 1..7")
    if [p["id"] for p in sorted(correct,key=lambda x:x["order"])] != seq:
        err(cid, "correct_sequence не совпадает с порядком order")
    ids = [p["id"] for p in pool]
    if len(ids) != len(set(ids)): err(cid, "дублирующиеся id в pool")
    for p in pool:
        if not p.get("text") or not p.get("explain"): err(cid, f"{p.get('id')}: пустой text/explain")

def main():
    if len(CASES) != 50: print(f"⚠ кейсов {len(CASES)}, цель 50")
    ids = [c["id"] for c in CASES]; oidx = [c["order_index"] for c in CASES]
    if len(set(ids)) != len(ids): raise SystemExit("дублирующиеся id кейсов")
    if len(set(oidx)) != len(oidx): raise SystemExit("дублирующиеся order_index")
    for c in CASES: validate(c)
    print(f"OK: {len(CASES)} кейсов валидны")

if __name__ == "__main__":
    main()
```

> Этот валидатор — обязательный гейт. Дополни его, если введёшь новые поля.
> Желательно добавить вызов в CI (по аналогии с TZ-1 blocking scope), но как минимум —
> прогонять локально перед каждым `seed_cases`.

## 10. Процесс посева и проверки (verify-before-done)

Локально (нативная дев-среда, пакетный менеджер `uv`):

```bash
cd apps/api
uv run python -m scripts.validate_cases        # 1. структура зелёная
uv run python -m scripts.seed_cases            # 2. посев в БД (DELETE+insert всех CASES)
```

Прогон логики кейса напрямую (без CSRF/HTTP), на каждый новый кейс хотя бы один проход:

```python
# пример: пройти верный путь и убедиться в 100%
# start → 5 верных ответов → outcome (stage1_score=50) → submit-order(s1..s7) → stage2_score=50
# и один НЕВЕРНЫЙ ответ, чтобы убедиться, что info-узел открывается и сходится вперёд.
```

Браузерная проверка (не на каждый, но на выборку и на все «сложные» D4–D5):
- Залогиниться тест-юзером (роль `manager`, `onboarding_completed=True` — иначе онбординг-тур
  будет перекидывать между страницами!).
- Открыть `/cases/<id>`, пройти оба этапа, увидеть 100% при верных ответах, проверить:
  сиреневую полоску, факт-карточки, экран info-разбора при неверном ответе, отсутствие
  «Маняши» и breadcrumb.
- Удалить тест-юзера после проверки.

**Чек-лист «готово» на кейс:**
1. Валидатор зелёный.
2. Верный путь даёт 50+50=100.
3. Хотя бы один неверный ответ ведёт в info и сходится к следующему вопросу.
4. Юридически достоверно (нормы/цифры сверены).
5. Нет ЮЛ-терминов как «правильных», нет Маняши.
