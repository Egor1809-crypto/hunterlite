## 0. Что просили (дословно, разобрано)
«Чтобы участвовать — нужно пройти курсы. Один курс выходит раз в неделю по четвергам в 19:00.
После просмотра открывается панель/таб, чтобы отметиться, что просмотрел. Есть полоска % прохождения.
Если не нажмёт галочку — нужно смотреть заново. Будет 3 галочки: посмотрел / не посмотрел.
Чтобы участвовать в чемпионате и пройти курс — нужно 100% просмотренности.»

Разбор по пунктам:
1. **Дрип-выдача**: уроки/курсы открываются по расписанию — еженедельно, четверг 19:00.
2. **Отметка просмотра**: после урока — панель с подтверждением «просмотрел».
3. **Прогресс-бар**: % прохождения курса.
4. **Без отметки — пересмотр**: урок не засчитан, пока пользователь явно не отметил.
5. **«3 галочки»**: ⚠️ неоднозначно — см. §6 вопрос 1.
6. **Гейт 100%**: участие в чемпионате требует 100% просмотренности обязательных курсов.

## 2. Модель данных (бэкенд, предложение)

### 2.1 Дрип-расписание
Якорь на курс: дата старта дрипа + еженедельный шаг (чт 19:00). Урок `i` (0-based) открывается:
`unlock_at(i) = course_drip_start + i недель`, всегда на четверг 19:00 (таймзона — §6 в.2).
- Где хранить `course_drip_start`: новая таблица `courses` (slug, title, drip_start_at, …) **или** конфиг-константа на курс. Рекоменд.: лёгкая таблица `courses` (чтобы менять даты без деплоя).

### 2.2 Прогресс
`course_lesson_progress`:
- `id`, `user_id` FK, `course_slug`, `lesson_index` (int), `watched_at` (timestamp), `created_at`.
- UNIQUE(`user_id`, `course_slug`, `lesson_index`).
- Наличие строки = урок отмечен просмотренным. (Состояние «не посмотрел» = отсутствие строки.)

> Список уроков остаётся во фронтовом `data.ts` (источник кол-ва/порядка). Бэкенд хранит только факт
> отметки по ключу (slug, index). Если решим перенести курсы в БД целиком — отдельный шаг.

## 3. API (FastAPI, auth)
- `GET /courses/progress` → по каждому курсу: `total_lessons`, `unlocked` (сколько открыто по дрипу сейчас),
  `watched` (отмечено), `percent` (watched/total), `next_unlock_at`, и по урокам — `locked|available|watched`.
- `POST /courses/{slug}/lessons/{index}/watched` → отметить просмотр. Гард: урок должен быть **открыт** по
  дрипу (нельзя отметить будущий). Идемпотентно.
- (опц.) `DELETE /courses/{slug}/lessons/{index}/watched` — снять отметку (если нужно «пересмотреть»).

## 4. Фронтенд UX
- Страница курса `/courses/[slug]`: каждый урок в одном из состояний:
  - **Заблокирован** (до `unlock_at`): плашка «Откроется в чт 19:00 · через N дн.» (счётчик).
  - **Доступен**: кнопка «Смотреть» (внешнее видео) + после возврата — панель/таб с **мини-проверкой**.
  - **Пройден**: галочка + дата (после успешной мини-проверки).
- **Полоска % прохождения** курса сверху (пройдено/всего).
- **Мини-проверка вместо самоклика** (решение §6): после видео — панель с **2-3 вопросами** по уроку.
  «Пройдено» = ответил верно (порог — §6). «3 галочки» = результаты 3 вопросов (✓/✗). Это честная
  проверка (нельзя накрутить, не посмотрев) и реальное обучение. Переиспользуем движок экзамена/квизов
  (`exam_item` / knowledge-квизы), не строим с нуля.
- 100% = все уроки курса пройдены (бейдж «Курс пройден»).

## 6. Решения и оставшиеся вопросы

### ✅ Принято 2026-06-10
- **Дрип-расписание:** глобальное — урок N открывается всем в один и тот же четверг 19:00 от даты старта
  курса (таймзона МСК, подтвердить если не так).
- **Обязательные курсы для чемпионата:** оба («Юр.аспекты» + «Эксперт.БФЛ»).
  - ⚠️ Обработка пустого курса: курс с 0 уроков считается «не активен» и **временно исключается из гейта**,
    пока в него не зальют уроки; как только появятся — автоматически входит в обязательные. Иначе условие
    было бы невыполнимо для всех. (Подтвердить такую трактовку.)
- **Объём:** поэтапно.
  - **Этап 1:** модель `course_lesson_progress` + API (`GET /courses/progress`, `POST …/watched`) +
    фронт: %-полоска и состояния уроков + отметка просмотра.
  - **Этап 2:** дрип-расписание (таблица `courses.drip_start_at`, блокировка будущих уроков, счётчики).
  - **Этап 3:** гейт 100% в `championship_qualification` (замена прокси-по-подписке) + правка Положения §2.1
    и интро Маняши под реальное условие.

### ✅ Принято 2026-06-10 (вторая итерация)
- **Механика «пройдено» = мини-проверка** (2-3 вопроса по уроку), НЕ самоклик-галочка. Защита от накрутки +
  реальное обучение. «3 галочки» = результаты вопросов. Переиспользуем движок экзамена/квизов.
- **Дрип = глобальный календарь с догоном.** Урок `i` разблокируется всем в момент `unlock_at(i)`
  (общий для всех). Урок **доступен** пользователю, если `now ≥ unlock_at(i)` — независимо от даты входа.
  → Поздно зашедший видит **все уже вышедшие уроки сразу** (догоняет: смотрит + проходит мини-проверку),
  будущие открываются по общему расписанию. Никого не отсекает.
- **Курс пройден** = все уроки курса пройдены (видео + мини-проверка). Гейт чемпионата = оба курса пройдены.

### ✅ Принято 2026-06-10 (третья итерация — финал)
- **Вопросы:** по **3 на урок**, генерирую сам по названию урока (черновик на вычитку; юр-точность —
  под последующую проверку юристом, как и Положение).
- **Порог:** **3/3 строго**, правильные ответы **не подсвечиваются**, **3 попытки**. После 3 неудачных —
  «пересмотрите урок» → повтор (попытки сбрасываются при повторном открытии урока).
- **Таймзона:** МСК.
- **Грейдинг — серверный:** правильные ответы хранятся ТОЛЬКО на бэкенде; фронт получает вопросы без
  ответов, шлёт выбор, сервер оценивает (3/3) и считает попытки. (Иначе ответы видны в devtools.)

### ⏳ Организационное (не код)
- Оператор задаёт дату старта дрипа так, чтобы последний урок вышел **до недели подсчёта** сезона.
- «Эксперт.БФЛ» пуст → вне гейта, пока не зальют уроки (потом авто-входит).
