# ТЗ: Переписать режим «Звонок» с нуля — простой стриминг-каскад

> **Аудитория:** агент-исполнитель, пишущий систему с нуля.
> **Статус:** утверждённый план. Основан на 2 deep-research отчётах (адверсариально
> проверено 24/25 + 22/25 фактов) + прямых замерах navy.api.
> **Принцип:** идеально простая, лёгкая, быстрая система. Каждая строка самописного
> «умного» VAD/шумового пола/one-turn-lock — это баг. Выбрасываем всё, ставим
> явный push-to-talk и стриминг-каскад.

---

## 0. Вердикт (что строим и почему)

**Строим: самосборный стриминг-каскад `STT → стриминг-LLM → посегментный TTS`
на navy.api, транспорт — FastAPI WebSocket, turn-taking — push-to-talk.**

Отвергнуто (с обоснованием):
- ❌ **ElevenLabs Conversational AI** (turnkey) — ресёрч НЕ нашёл подтверждённых
  данных по custom-LLM endpoint / транскриптам / цене. Непроверяемый риск + теряем
  контроль над персоной и оценкой. Если очень хочется — это отдельная задача-спайк.
- ❌ **Pipecat / LiveKit** (фреймворк) — overkill для одного линейного диалога 1:1
  без ветвления. Их дефолтные пайплайны на Deepgram/Cartesia, под navy всё равно
  перенастраивать. Ядро нашего цикла = ~100 строк.
- ❌ **Speech-to-speech / Realtime** (Gemini Live / OpenAI Realtime) — на navy не
  подтверждён; теряем транскрипты, нужные для оценки в конце.
- ❌ **Серверный VAD / семантический turn-detector** — push-to-talk делает границы
  реплики однозначными и убирает весь класс проблем (это и был источник «молчания»
  и «ИИ отвечает на стартовую реплику»). [voiceaiandvoiceagents.com, Inworld]

**Целевая латентность:** реплика→первый звук ИИ **~3с** (узкое место — LLM TTFT
на navy ~1.9с, его не обойти на этом стеке). Сейчас 7-22с. Маскируется филлером
(см. §5), так что ВОСПРИНИМАЕТСЯ как ~1с.

## 1. Прямые замеры navy.api (ground truth — НЕ перепроверять)

| Стадия | Endpoint | Параметры | Замер |
|---|---|---|---|
| **STT** | `POST /v1/audio/transcriptions` | `model=gpt-4o-transcribe`, multipart `file` (webm), `language=ru`, `response_format=json` | ~1.3с / реплика |
| **LLM** | `POST /v1/chat/completions` | `model=claude-haiku-4.5`, `stream=true` | TTFT ~1.9с ✅ стриминг работает |
| **TTS** | `POST /v1/audio/speech` | `model=eleven_flash_v2_5`, `voice=<id>`, `input=<предложение>` → MP3 | ~0.5-0.7с / короткое предложение ✅ |

- База: `https://api.navy/v1`, заголовок `Authorization: Bearer <NAVY_LLM_API_KEY>`
  (в коде — `settings.local_llm_api_key`, base — `settings.local_llm_url`).
- ⚠️ `gpt-4o-transcribe` НЕ принимает `response_format=verbose_json` (только `json`/`text`).
- ⚠️ navy нестабилен: разовые спайки до 11-139с. Поэтому ОБЯЗАТЕЛЕН таймаут на
  каждый вызов + graceful-фоллбэк (филлер/«повторите, пожалуйста»).
- TTS отдаёт **готовый MP3 целиком** (batch). Стриминг достигается на уровне
  **предложений**: каждое предложение = отдельный ~0.5с вызов → очередь
  воспроизведения. Chunked-стрим ВНУТРИ файла не нужен.

**Русские голоса ElevenLabs (voice_id для `/v1/audio/speech`):**
- `ouyTiWqmHA5WI5bbX7zj` — мужской
- `AB9XsbSA4eLG12t2myjN` — женский
- `ETBmMkYUh8i2exSl2h3P` — женский (запасной)

Голос выбирается по полу персоны (`client_gender`).

## 2. Архитектура (поток одного хода)

```
[Браузер]                         [FastAPI WS]                    [navy.api]
1. Зажал «Говорить»  ──────────►  (recording)
2. Отпустил кнопку
   → MediaRecorder stop
   → webm Blob       ──ws audio──►  3. СРАЗУ играет филлер  ◄── (предзагружен)
                     ◄─filler────   (кэш MP3 «Хм, секунду…»)
                                    4. STT (webm→текст)      ──► /v1/audio/transcriptions
                     ◄─transcript─   (показать что распознал)
                                    5. LLM stream=true       ──► /v1/chat/completions
                                       буфер токенов,
                                       детект границы «.!?»
                                    6. готово предложение #1 ──► /v1/audio/speech (TTS)
                     ◄─audio #1───   отправить MP3 #1
                                    7. предложение #2 → TTS  ──► /v1/audio/speech
                     ◄─audio #2───   отправить MP3 #2
   8. Очередь playback             ...
                     ◄─turn_end───   (можно говорить снова)
```

Ключ к скорости: пока пользователь слышит **филлер** (шаг 3) и потом **первое
предложение** (шаг 6), LLM ещё генерит остаток. Воспринимаемая задержка ≈ филлер
(мгновенно) → первое предложение (~3с), а не вся реплица.

## 4. Бэкенд — новый WS-обработчик `app/ws/call.py`

Один WebSocket-эндпоинт. Состояние сессии в памяти соединения (история сообщений,
персона, голос). **Без** общего training-WS, без эмоциональной FSM.

### 4.1 WS-протокол (JSON, кроме бинарного аудио)

**Клиент → сервер:**
```jsonc
{ "type": "start", "session_id": "...", "persona_slug": "..." }   // инициализация
// затем бинарный кадр: webm-аудио всей реплики (после отпускания кнопки)
{ "type": "end_call" }                                            // завершить → оценка
```

**Сервер → клиент:**
```jsonc
{ "type": "ready", "client_name": "...", "client_card": {...} }   // персона загружена
{ "type": "filler", "audio_b64": "...", "mime": "audio/mpeg" }    // мгновенный филлер
{ "type": "transcript", "role": "user", "text": "..." }           // что распознал STT
{ "type": "sentence", "index": 0, "text": "...", "audio_b64": "..." } // предложение+аудио
{ "type": "turn_end" }                                            // ход ИИ завершён
{ "type": "client_hangup", "text": "..." }                        // персона сама закончила
{ "type": "score", "result": {...} }                              // финальная оценка
{ "type": "error", "message": "..." }
```

### 4.2 Цикл обработки хода (псевдокод — ядро ~100 строк)

```python
async def handle_user_turn(ws, state, webm_bytes):
    # 1. филлер СРАЗУ (пока STT+LLM думают) — из предзагруженного кэша
    await ws.send_json({"type": "filler", "audio_b64": random_filler(state.gender)})

    # 2. STT (таймаут 8с, фоллбэк — попросить повторить)
    text = await stt_transcribe(webm_bytes, model="gpt-4o-transcribe", language="ru")
    if not text.strip() or is_junk(text):       # «Tuo», «谢谢», «.» — дроп
        await ws.send_json({"type":"sentence","index":0,"text":"Извините, не расслышал, повторите?","audio_b64": tts(...)})
        await ws.send_json({"type":"turn_end"}); return
    state.history.append({"role":"user","content":text})
    await ws.send_json({"type":"transcript","role":"user","text":text})

    # 3. LLM stream=true + посегментный TTS
    buf, idx = "", 0
    async for token in llm_stream(state.history, model="claude-haiku-4.5", system=state.system_prompt):
        buf += token
        sentence, buf = pop_complete_sentence(buf)   # детект .!? + защита от Dr./3.14
        if sentence:
            audio = await tts(sentence, voice=state.voice)     # /v1/audio/speech, ~0.5с
            await ws.send_json({"type":"sentence","index":idx,"text":sentence,"audio_b64":audio})
            state.assistant_buf += sentence; idx += 1
    if buf.strip():                                   # хвост без финальной точки
        audio = await tts(buf, voice=state.voice)
        await ws.send_json({"type":"sentence","index":idx,"text":buf,"audio_b64":audio})
        state.assistant_buf += buf
    state.history.append({"role":"assistant","content":state.assistant_buf})

    # 4. персона решила закончить? (текст содержит маркер прощания)
    if wants_hangup(state.assistant_buf):
        await ws.send_json({"type":"client_hangup","text":state.assistant_buf})
    else:
        await ws.send_json({"type":"turn_end"})
```

`pop_complete_sentence`: ищем `[.!?…]` + followed-by-space/EOL; НЕ резать на
`Dr.`/`т.е.`/десятичных (`3.14`); минимум ~10 символов перед flush.
[AssemblyAI: «buffer tokens, watch for sentence boundaries, flush each completed
sentence to TTS the instant it's ready»]

### 4.3 Точные вызовы navy

```python
# STT
files = {"file": ("turn.webm", webm_bytes, "audio/webm")}
data  = {"model": "gpt-4o-transcribe", "language": "ru", "response_format": "json"}
# POST {local_llm_url}/audio/transcriptions  (multipart)  → {"text": "..."}

# LLM (стриминг)
{ "model": "claude-haiku-4.5", "stream": true, "max_tokens": 220, "temperature": 0.8,
  "messages": [{"role":"system","content": system_prompt}, *history] }
# POST {local_llm_url}/chat/completions  → SSE, читать data: чанки

# TTS (на каждое предложение)
{ "model": "eleven_flash_v2_5", "voice": voice_id, "input": sentence }
# POST {local_llm_url}/audio/speech  → bytes (audio/mpeg) → base64
```
`max_tokens=220`: реплики должника КОРОТКИЕ (1-3 предложения) — это и быстрее, и
естественнее. Длинные монологи = баг.

## 5. Маскировка латентности (3 проверенных приёма)

1. **Посегментный TTS** (§4.2) — первое предложение озвучивается, пока LLM генерит
   второе. Сокращает воспринимаемую задержку на ~60-80% для реплик >2 предложений.
2. **Филлеры** (главный приём, peer-reviewed arXiv 2507.22352): сразу после
   отпускания кнопки играть короткую разговорную реплику-филлер, пока STT+LLM
   думают. **Заранее** синтезировать набор (TTS один раз при старте сессии или
   статичные файлы) и кэшировать:
   - муж: «Так… секунду.», «Хм, ну…», «Подождите-ка.»
   - жен: «Так… минутку.», «Хм…», «Сейчас-сейчас.»
   Искусственные спиннеры НЕ помогают — только голосовой филлер. Филлер играется
   ПЕРЕД первым предложением; не накладывать (очередь playback на FE сама
   упорядочит: filler → sentence#0 → …).
3. **Параллелизация** — филлер уходит мгновенно (шаг 1) НЕ дожидаясь STT.
   (Стриминг-STT для экономии 200-400мс на navy НЕ применим — нет WebSocket-STT,
   эту оптимизацию из плана ИСКЛЮЧИТЬ.)

## 6. Скриптовая логика персоны (БЕЗ галлюцинаций)

Причина «Ирины из Саратова» — досье НЕ попадало в промпт. Фикс: структурированный
**Agent Playbook** system-prompt (9 секций, грунтинг фактами), строится из
`persona_brief`/`client_card`. НЕ хардкод-FSM. [arXiv 2509.04871]

```
1. РОЛЬ: Ты — {full_name}, {age} лет, {city}. НЕ ассистент, НЕ ИИ. Реальный человек-должник.
2. ХАРАКТЕР/ТОН: {emotion_preset → тревожный/злой/...}. Говоришь коротко, по-телефонному.
3. ТВОЁ ДОСЬЕ (ЕДИНСТВЕННЫЙ источник фактов — НЕ выдумывай сверх него):
   {persona_brief дословно: профессия, состав долга, кредиторы, семья, имущество, статьи}
4. ХОД РАЗГОВОРА: тебе звонит юрист по банкротству. Сначала недоверие («откуда мой
   номер?»), потом — по мере убеждения — раскрываешься.
5. ВОЗРАЖЕНИЯ: сомневаешься в законности, боишься потерять имущество, спрашиваешь цену.
6. ГРУНТИНГ: если факта нет в досье — отвечай уклончиво, НЕ изобретай (город, возраст,
   работодателя, суммы). НИКОГДА не называй данных, которых нет в досье.
7. ФОРМАТ: 1-3 коротких предложения. Разговорный русский. БЕЗ markdown, БЕЗ хештегов,
   БЕЗ ремарок в скобках/звёздочках.
8. ЗАВЕРШЕНИЕ: если юрист груб/непрофессионален — можешь закончить разговор фразой
   прощания (тогда добавь в конце маркер «[КЛАДУ ТРУБКУ]»).
9. ПРИМЕР (коротко, НЕ копируй дословно): Юрист: «Здравствуйте, я по вашим долгам».
   Ты: «Да? А откуда у вас мой номер?»
```

- Грунтинг фактами + «не изобретай» = нет галлюцинаций персоны.
- Пример КОРОТКИЙ, чтобы модель не «попугайничала».
- `wants_hangup()` детектит маркер `[КЛАДУ ТРУБКУ]` (вырезать из озвучки).

## 7. Оценка в конце (rubric LLM-судья)

При `end_call` — весь транскрипт (короткий звонок тривиально влезает в контекст)
оценивается LLM-судьёй. [EvidentlyAI, Azure AI Foundry, Hamel Husain]

**Правила (важно):**
- НЕ просить балл «73 vs 82» — LLM не калиброваны для высокоточных оценок.
- Каждый аспект — ОТДЕЛЬНЫЙ оценщик (один вызов на критерий) для фокуса/точности.
- Критерии — бинарные / шкала 0-5 (0-5 лучше всего коррелирует с людьми).
- Агрегировать ДЕТЕРМИНИРОВАННО (код, не LLM).

**Рубрика (адаптировать под БФЛ-звонок):**
| Критерий | Тип |
|---|---|
| Представился и обозначил цель | 0/1 |
| Установил контакт, не давил грубо | 0-5 |
| Выявил состав/причину долга | 0/1 |
| Отработал возражения клиента | 0-5 |
| Предложил корректное решение (банкротство/реструктуризация) | 0-5 |
| Не нарушил закон / не дал ложных гарантий | 0/1 |

- Использовать существующую `scoring.py`-инфраструктуру (контентный guard на
  пустой/мусорный звонок: <2 реальных реплик юриста → 0 баллов, как в чате).
- Модель судьи: `deepseek-v4-pro` или `gpt-4o-mini` (точность важнее скорости — не
  в real-time).

## 8. Фронтенд — новый `call/page.tsx` (≤ 400 строк вместо 2157)

- **Push-to-talk:** одна большая кнопка «Говорить» (hold = `onPointerDown`
  start `MediaRecorder`, release = `onPointerUp` stop → отправить Blob по WS).
  Опционально toggle-режим (клик старт / клик стоп) для удобства. **Никакого VAD.**
- **Захват:** `MediaRecorder(stream, {mimeType:"audio/webm;codecs=opus"})`, один Blob
  на реплику. (Это уже доказанно работает в проекте — взять из чата/call как основу.)
- **Аудио-очередь:** массив поступающих MP3 (filler → sentence#0 → sentence#1 → …),
  играть последовательно через один `<audio>`/`AudioContext` (новый только после
  `ended` предыдущего). На `client_hangup`/`end_call` — стоп + дренаж очереди.
- **UI-статус:** «Говорите» / «Обрабатываю…» / «Говорит {Имя}». Без «эмоция-пилюль».
- **Unlock автоплея:** один клик «Ответить» на стартовом экране (НЕ 3-векторный цирк).
- **Завершение:** кнопка «Завершить» → `{type:"end_call"}` → ждём `score` → `/results`.
