## 0. Краткая суть и ГЛАВНОЕ предупреждение

Делаем **самую маркетинговую страницу платформы** — чемпионат/розыгрыш в стиле apple.com:
крупная типографика, белый воздух, один акцент, scroll-сторителлинг призов (MacBook Air 13" M4,
iPhone 15, AirPods 4). Страница живёт в **двух местах** и закольцована:
- на **лендинге** — новая вкладка «Чемпионат» (+ панель над переключателем темы);
- на **платформе** — `/certificate/contest` (уже существует) + дубль-панель над `ThemeToggle` в сайдбаре.

### 🔴 Критический юридический риск (РЕШЕНИЕ ПРИНЯТО: рандомайзер) — обязательные меры
Заказчик выбрал **рандомайзер (жребий)** для определения победителей. Ресёрч по праву РФ предупреждает:
**жребий = элемент случайности → риск переквалификации акции в ЛОТЕРЕЮ**, а стимулирующие лотереи с
01.07.2014 (ФЗ-416 от 28.12.2013) выведены из-под ФЗ-138 и по сути запрещены к самостоятельному
проведению. Чтобы при этом остаться **стимулирующим мероприятием по ст.9 ФЗ-38** и максимально снизить
риск, реализуем по такой защищённой схеме:

1. **Розыгрыш проводится ТОЛЬКО среди квалифицированных участников** (выполнивших объективные условия:
   аттестация ≥88%, платные курсы, подписка, отзыв). Жребий — финальный шаг среди уже отобранного пула.
2. **Никакой «платы за билет»** — участие бесплатно; платные курсы покупаются как часть продукта, а не как
   взнос за участие (правильная формулировка против «скрытой лотереи», см. §9 п.4).
3. **Защищённый рандомайзер с фиксацией результата** — сертифицированный/верифицируемый сервис
   (random.org с подписанным результатом / лототрон / нотариальная фиксация) + **видеозапись и скрин**
   момента розыгрыша, публикуемые публично. Конкретный сервис — нужен отдельный мини-ресёрч (PILLAR C
   не был исследован), см. §11 п.7.
4. **Ранги остаются** (золото/серебро/бронза) — это просто привязка призов к 3 вытянутым победителям
   (1-й вытянут → MacBook, 2-й → iPhone, 3-й → AirPods). Пьедестал-графику убираем (как и просили).
5. **🔴 Обязательная юр-виза Положения до публичного запуска** — при выбранной механике это не опционально.

> Если на этапе юр-визы юрист забракует жребий — fallback на объективный рейтинг (по баллам) остаётся
> запасным вариантом, код лидерборда это поддержит.

## 2. Логика периодов чемпионата

### 2.1 Два сезона в год (фикс из ТЗ)
| Сезон | Старт | Финиш активной фазы | Неделя подсчёта |
|---|---|---|---|
| **Зимне-весенний** | 1 декабря | последняя неделя весны − 1 | последняя неделя весны |
| **Летне-осенний** | 1 июня | предпоследняя неделя осени | последняя неделя осени |

> Сейчас открыт **Чемпионат №1** (нумерация сквозная). По дате (10.06.2026) активен летне-осенний сезон,
> старт 01.06.2026.

### 2.2 Состояния сезона (FSM)
`upcoming` (анонс, до старта) → `active` (идёт приём результатов) → `tallying` (неделя подсчёта, лидерборд
заморожен) → `finished` (победители объявлены, уходит в Историю).

UI меняется по состоянию: счётчик «до старта» / «до конца» / «идёт подсчёт» / «итоги».

### 2.3 Вкладка «История победителей»
Отдельная вкладка/секция. Сейчас **пустая** (первый чемпионат) — показываем красивый empty-state
(«Здесь появятся имена первых чемпионов»). После `finished` сезоны рендерятся сюда из БД (реальная история).

## 3. Модель данных и API

### 3.1 Новые таблицы (Alembic-миграции; конвенция `YYYYMMDD_NNN_*`)
- **`championships`** — сезон: `id`, `number` (сквозной №), `season_type` (winter_spring|summer_autumn),
  `title`, `starts_at`, `tally_starts_at`, `ends_at`, `status` (FSM), `prize_fund` (JSONB: список призов
  с названием/стоимостью/изображением/рангом), `created_at`.
- **`championship_entries`** — заявка/участие: `id`, `championship_id` FK, `user_id` FK, `status`
  (`enrolled|qualified|disqualified`), `score` (агрегированный балл по критериям), `metrics` (JSONB:
  exam_score, courses_done, lessons_done, review_left, speed_days), `created_at`, `updated_at`.
  UNIQUE(`championship_id`,`user_id`).
- **`championship_winners`** — итоги: `id`, `championship_id` FK, `user_id` FK, `rank` (1..N),
  `prize` (название), `published_name`, `publish_consent` (bool, ст.10.1), `created_at`.

> `Review` (таблица `reviews`) **уже существует** (миграции есть) — отдельная история, см. §6.

### 3.2 API (FastAPI; `apps/api/app/api/championship.py`, регистрация в `router.py`)
- `GET /championship/current` — активный сезон + его параметры/состояние/призы.
- `GET /championship/{id}/leaderboard` — объективный лидерборд (топ-N по `score`), для платформы.
- `GET /championship/winners` — история победителей (для вкладки «История»).
- `POST /championship/enroll` — подать заявку (auth; проверка критериев).
- `GET /championship/me` — мой статус/прогресс по критериям.
- Бэкенд-джоб: пересчёт `score`/`status` по событиям (экзамен сдан, курс пройден, отзыв оставлен).

### 3.3 Квалификация и розыгрыш (механика: рандомайзер среди квалифицированных)
**Квалификация** (объективный gate, метрики `championship_entries.metrics`): аттестация ≥88%, полнота
курсов (Юр.аспекты + Эксперт.БФЛ), подписка, факт оставленного отзыва → `status=qualified`.
**Розыгрыш:** среди всех `qualified` сторонним рандомайзером тянутся 3 победителя; порядок вытягивания →
ранг → приз (1→MacBook, 2→iPhone, 3→AirPods). Результат фиксируется (видео/скрин/подпись) и пишется в
`championship_winners`. Лидерборд на странице показывает **пул квалифицированных** (для вовлечённости можно
сортировать по активности), но он НЕ определяет победителя — победитель определяется жребием.
> Запасной режим (если юрист забракует жребий): победители = топ по баллу. Код поддерживает оба режима
> через поле `championships.winner_mode` (`draw` | `ranking`).

## 4. Дизайн в стиле Apple (scroll-сторителлинг)

> Стек проекта: React + **Framer Motion** (в коде уже `motion`) + Tailwind. Все техники ниже —
> подтверждены первичной докой Motion (motion.dev). **Это сознательное отклонение от malvah-стиля**
> (см. §9 — решение №2).

### 4.1 Дизайн-токены (Apple HIG)
- Типографика: SF Pro-подобный стек (system-ui/`-apple-system`), очень крупные заголовки
  `clamp(40px, 7vw, 96px)`, плотный tracking на больших, тонкие веса на подзаголовках.
- Цвет: near-black `#1d1d1f` текст на белом/светло-сером; **один акцент** — синяя «pill»-кнопка
  (`#0071e3`-класс). Много воздуха, ритм секций ~`100vh`.
- На тёмной теме платформы — инвертируем (тёмный фон + тот же акцент), сохраняя Apple-композицию.

### 4.2 Раскадровка страницы (сверху вниз, с эффектами)
1. **Hero** — крупный заголовок «Чемпионат сезона», подзаголовок, счётчик до конца сезона,
   плавающий первый приз. Эффект: hero-типографика масштабируется/тает при скролле
   (`useScroll` → `useTransform(scrollYProgress,[0,1],[1,0.92])` + opacity).
2. **Sticky-сторителлинг призов** (3 экрана) — каждый приз «приезжает» и фиксируется:
   tall-контейнер + `position:sticky top:0` + `useTransform` для входа изображения, параллакс фона
   (фон медленнее переднего плана: разные output-диапазоны). Опционально — canvas image-sequence
   (Apple-приём «покадровая прокрутка») для вращения девайса; это отдельный набор кадров (тяжелее,
   фаза 2).
3. **«Как участвовать»** — 3–4 шага (купить курс → пройти → оставить отзыв → попасть в зачёт),
   reveal по одному (`whileInView` + `viewport={{once:true}}`, stagger).
4. **Пул квалифицированных + розыгрыш** — список участников, прошедших условия; после розыгрыша —
   3 победителя с рангами золото/серебро/бронза (без пьедестала) и привязкой к призам.
5. **История победителей** — вкладка (пустой стейт сейчас).
6. **FAQ + футер** — документы (Положение, Политика ПДн, Согласие), ТГ-бот, реквизиты организатора.
7. Сквозной **scroll-progress bar** сверху (`style={{scaleX: scrollYProgress}}`).

### 4.3 Конкретные приёмы Motion (из ресёрча)
- `const {scrollYProgress} = useScroll()` → 0→1.
- Прогресс-бар: `<motion.div style={{scaleX: scrollYProgress}}/>`.
- Per-section: `useScroll({target: ref, offset: ['start end','end end']})`.
- Reveal/blur: `useTransform(p,[0,1],['blur(10px)','blur(0px)'])`, opacity, translate.
- Параллакс: фон `[0,0.5]`, передний план `[0,2]`.
- Sticky-горизонталь: tall `~300vh` + `position:sticky` + `x = useTransform(p,[0,1],['0%','-75%'])`.
- Производительность: `prefers-reduced-motion` (отключать анимации), lazy-загрузка кадров.

## 9. Белые пятна, противоречия и риски (из ресёрча — обязательно к учёту)
1. **Рандомайзер vs закон** (см. §0) — РЕШЕНО: оставляем жребий, но ТОЛЬКО среди квалифицированных, без
   платы за билет, с фиксацией и **обязательной юр-визой**. Остаётся главным юр-риском проекта.
2. **Стиль Apple vs malvah** — РЕШЕНО: «Apple-композиция + наш фирменный акцент» (не полный Apple,
   не строго malvah). Эта страница — сознательное исключение из malvah-инварианта.
3. **НДФЛ-агент для неденежных Apple-призов** — организатор не удерживает; победитель сам платит.
   Обязательно прописать в Положении и в UI («с приза уплачивается НДФЛ 35%»), иначе споры.
4. **«Де-факто платное участие»** — формулировка «участие платное, т.к. нужны платные курсы» юридически
   рискованна (может читаться как скрытая лотерея/плата за билет). Правильная рамка: участие **бесплатно**,
   но доступ к зачёту даёт прохождение платных курсов как часть продукта. Переформулировать в Положении.
5. **Отдельное согласие ст.10.1** на публикацию имени/фото — отдельный чекбокс, не смешивать с общим.
6. **Один мастер-Положение vs по сезону** — рекомендуем мастер-Положение + приложение с призами на сезон
   и отдельный цикл уведомления ФНС на каждый сезон. ❗ решение №4.
7. **Актуальность ставок** — поправки 152-ФЗ от 01.09.2025 и налоговая реформа 2025 ужесточили правила;
   перед каждым запуском сезона перепроверять 35%/4000₽ и тексты согласий у юриста.
8. **Сектор-специфика** — ООО (банкротная тематика), продающее курсы, перед публичным запуском желательно
   получить юр-визу на Положение (не блокирует разработку, но фиксируем как обязательный шаг до go-live).
9. **PILLAR C не закрыт ресёрчем** — конкретные рандомайзеры не исследованы (и не нужны, если переходим на
