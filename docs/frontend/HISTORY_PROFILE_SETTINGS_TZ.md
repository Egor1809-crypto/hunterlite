## 0. Контекст и цель
Три страницы личного кабинета. Заказчик: **полный объём** —
- **/история**: реальные отчёты по тесту/кейсам/конструктору/экзамену + Маняша
  (разбор по клику + недельный отчёт) + перенести сюда 180-дн. активность; апгрейд.
- **/профиль**: **удалить целиком**; перенести пароль → настройки, активность → история,
  TG-подключение → настройки.
- **/настройки**: убрать мёртвые артефакты; добавить TG-бота и смену пароля.

**Важно (граница продукта):** «Тесты» (`TestWorldMap`/`KnowledgeQuizSession`) — это
**продукт (kept)**. Их отчёты в истории **оставляем** (НЕ путать с удалённым PvP-квизом).

## 1. /история (`apps/web/src/app/history/page.tsx`)

### 1.1 Сейчас (факты)
- `GET /training/history?limit=50` (`apps/api/app/api/training.py:2017`) → `kind ∈
  {session, story, crm_client, quiz}`. `session/story` несут реальные `score_*`; `quiz`
  = результаты теста (KnowledgeQuizSession) — **оставляем**.
- НЕТ отчётов: **экзамен** (`api/exams.py:377` отдельно), **кейсы** (`api/cases.py:437`
  `CaseAttempt`), **конструктор** (свои сессии). Маняши в истории нет. 180-дн. хитмап —
  в /профиль (`ActivityHeatmap days=180`).

### 1.2 Сделать
1. **Единая история со всеми реальными отчётами.** Расширить `GET /training/history`
   (или новый `GET /history/unified`) агрегацией: тренировки/конструктор-сессии +
   `CaseAttempt` (kind `case`) + `ExamAttempt` (kind `exam`) + тест (kind `quiz`).
   Новые `kind`: `case`, `exam`. Каждая запись — с **реальными метриками** (баллы/
   подбаллы/проценты/прошёл-нет), кликом ведёт на свой результат (`/results/*`,
   `/exam/certificate/*`, кейс-разбор).
2. **Маняша: разбор по клику.** Новый эндпоинт `GET /history/{kind}/{id}/explain` →
   через `knowledge_assistant`/`exam_grader`-контекст: Маняша делает **подробный отчёт**
   по слабым местам конкретного действия (что упустил, что сделать лучше; заземление на
   ФЗ-127). Возврат: текст + ссылки на чанки. UI: панель/раскрытие у записи.
3. **Маняша: недельный отчёт.** Бэк **почти готов** — `GET /dashboard/weekly-report`
   (`apps/api/app/api/dashboard.py:741`, есть `report_text`/`weak_points`/`recommendations`).
   На /история — блок «Итоги недели от Маняши»: Маняша анализирует все пройденные за
   неделю задания (тест/кейсы/конструктор/экзамен) и даёт сводку. **Расширить агрегацию**
   weekly-report экзаменом/кейсами (сейчас их там нет).
4. **Перенести 180-дн. активность** из /профиль (`ActivityHeatmap`,
   `GET /users/me/activity?days=180`) — компонент самодостаточный, перенос ~импорт.
5. **Апгрейд истории (визуал+функция):** реально работающие панели/отчёты вместо
   старых заглушек; компоненты в единой дизайн-системе (см. §6). Отчёты по
   тесту/конструктору/экзамену должны показывать **реальные показатели и учебную
   пользу**, а не пустышки.

### 1.3 Контракт (новый/изменённый)
```
GET /history/unified?limit=&offset=  → [{kind: session|story|case|exam|quiz, id,
     date, title, metrics:{...}, deep_link}]
GET /history/{kind}/{id}/explain     → {report_text, weak_points[], sources[]}  (Маняша)
GET /dashboard/weekly-report         → расширить агрегацию exam+case
```

## 4. Маняша: доступ к данным (архитектура)
Текущий `ManyashaChat` бьёт в Next-route `/api/chat` (**без auth/доступа к данным
пользователя**). Для отчётов по метрикам Маняша должна ходить в **FastAPI-бэк с токеном
сессии** (эндпоинты §1.3), а не в stateless Next-route. Переиспользовать персону Маняши;
данные (баллы/слабые места/неделя) берёт бэк. (Связано с ТЗ-3 — общий Маняша-бэкенд.)

## 5. Обработка ошибок / edge cases
- LLM недоступен → Маняша-разбор/недельный: graceful «отчёт временно недоступен»,
  страница работает.
- /профиль удалён → все ссылки ведут на /настройки или /история; глубокие ссылки на
  `/profile` → редирект, не 404.
- Перенос пароля — **обязателен** (иначе пользователь теряет доступ к смене пароля).
- `tsc --noEmit` чисто после удаления профиля/мёртвых настроек (нет висячих импортов).

## 6. Фронт-дизайн (отдельная фаза «по команде» — НЕ начинать без команды заказчика)
Использовать `/plan`: сначала исследовать, потом реализовать; **код не трогать до
утверждённого плана**. Открыть и доскроллить в реальном браузере оба референса —
`https://www.malvah.co/projects` и `https://www.abstract.com/` — выписать паттерны:
оформление панелей/блоков, типографику и иерархию, сетку и воздух, tone of voice
(хедеры, копирайт кнопок). Затем аудит текущего UI экран за экраном по 6 пунктам:
отступы/сетка/выравнивание; пустые/недоработанные блоки; компонентная система;
иерархия шрифтов/цветов; расположение панелей/кнопок; UX-механика каждой детали (что
делает кнопка, как реагирует, куда ведёт). План правок компонент за компонентом →
после утверждения реализация с проверкой в браузере (`verify`/`run`), итерируя
«посмотрел → поправил». Единая компонентная система (общие токены отступов/типографики/
цвета). В конце — состязательное ревью: рассинхрон выравнивания, «мёртвые» кнопки,
слабый копирайт — исправить. Прорабатывать и текст на каждой панели.
